## 一、模块概述

### 1.1 模块定位

商品看板是「全渠道销售综合看板」系统的子模块，支持 **SPU 维度** 和 **SKU 维度** 两种视图模式切换，帮助运营和供应链团队在不同粒度下快速掌握商品的销售表现、退款健康度及库存状况。

### 1.2 核心价值

- **双维度切换**：支持 SPU 维度（按商品聚合）和 SKU 维度（按颜色/规格分解）灵活切换，满足不同分析场景
- **快速搜索**：支持多条件组合搜索，SPU 维度可按抖音ID、货号搜索，SKU 维度可按商家编码搜索
- **下钻分析**：从 SPU 维度可点击进入 SKU 维度，查看同款商品各颜色的详细数据
- **全渠道对比**：同一商品在抖音、天猫、微信各渠道的销售和退款一目了然
- **退款预警**：通过颜色标记快速识别退款率异常的 SKU
- **库存联动**：现货库存 + 采购在途，辅助供应链决策

### 1.3 目标用户

| 角色 | 使用场景 |
|------|----------|
| 运营经理 | 监控 SKU 销售排名、退款率趋势，优化选品和定价策略 |
| 供应链专员 | 关注库存预警，安排补货和采购计划 |
| 渠道运营 | 对比各渠道表现，搜索特定商品跟进，调整投放策略 |
| 数据分析师 | 导出数据做深度分析 |

### 1.4 术语说明

| 术语 | 说明 |
|------|------|
| SPU | Standard Product Unit，商品聚合层，同一款商品（如"皮荔枝纹水桶手提包"）的所有颜色/规格共享同一个 SPU，关联抖音ID和货号 |
| SKU | Stock Keeping Unit，库存单元层，按每个颜色/规格作为独立行，关联商家编码 |
| 抖音ID | 抖音平台商品列表ID（数字串），SPU 级别唯一 |
| 货号 | 商品内部编号（格式如 B26ADxxx），SPU 级别唯一 |
| 商家编码 | 商家侧 SKU 编号（格式如 B26AYxxxxx），SKU 级别唯一 |

### 2.4 维度切换与下钻

#### 2.4.1 维度切换

- **切换方式**：点击顶部 Tab 切换「SPU维度」↔「SKU维度」
- **默认状态**：首次进入商品看板默认显示 **SPU 维度**
- **状态保持**：切换维度后，店铺筛选、时间周期等条件保持，搜索条件清空
- **URL 同步**：维度状态同步到 URL 参数，支持刷新后保持当前维度

#### 2.4.2 SPU 下钻到 SKU

- **触发方式**：点击 SPU 行任意位置（除展开图标外）
- **跳转效果**：页面切换至 SKU 维度，并自动填充该 SPU 下所有 SKU 的商家编码到搜索框
- **面包屑导航**：SKU 维度页面显示面包屑「商品看板 > SPU维度 > SKU维度」，点击可返回 SPU 维度
- **数据范围**：下钻后仅显示该 SPU 关联的 SKU 数据

### 3.5 退款率颜色阈值规则（基于历史均值动态校准）

退款率通过圆角标签（badge）展示，颜色阈值基于**历史均值动态计算**，而非固定数值。

#### 计算逻辑

系统每日自动计算最近 30 天的历史均值作为基准：

```
历史均值 = Σ(每日退款率) / 30 天

阈值计算：
- 预警下限 = 历史均值 × 0.9
- 预警上限 = 历史均值 × 1.3
```

#### 颜色判定规则

| 条件 | 背景色 | 文字色 | 含义 |
|------|--------|--------|------|
| ≤ 预警下限 | 浅绿 bg-emerald-50 | 绿色 text-emerald-600 | 正常（优于历史均值） |
| 预警下限 < rate ≤ 预警上限 | 浅黄 bg-amber-50 | 黄色 text-amber-600 | 预警（接近或略高于历史均值） |
| > 预警上限 | 浅红 bg-rose-50 | 红色 text-rose-600 | 异常（显著高于历史均值） |

#### 各退款率类型独立计算

| 退款率类型 | 计算基准 | 说明 |
|------------|----------|------|
| 总退款率 | 最近30天总退款率均值 | 售前+售后退款率之和的历史均值 |
| 现货退款率 | 最近30天现货退款率均值 | 仅现货订单退款率的历史均值 |
| 预售退款率 | 最近30天预售退款率均值 | 仅预售订单退款率的历史均值 |

#### 示例

假设最近 30 天总退款率均值为 20%：
- 预警下限 = 20% × 0.9 = 18%
- 预警上限 = 20% × 1.3 = 26%

则判定规则为：
| 当前退款率 | 颜色 | 含义 |
|------------|------|------|
| ≤ 18% | 绿色 | 优于历史平均水平 |
| 18% ~ 26% | 黄色 | 接近历史平均水平 |
| > 26% | 红色 | 显著高于历史平均水平，需关注 |

> **注意**：历史均值每日凌晨自动更新，阈值随之动态调整。

## 九、数据来源与计算规则

### 9.1 数据来源

| 数据类别 | 数据来源系统 | 说明 |
|----------|-------------|------|
| 销售金额/件数 | 订单系统 | 按下单时间统计，含所有状态订单 |
| 实付金额/件数 | 订单系统 | 扣除已退款订单后的净额 |
| 售前退款率 | 订单系统 | 发货前退款订单数 / 总订单数 × 100% |
| 售后退款率 | 订单系统 | 发货后退款订单数 / 总订单数 × 100% |
| 现货退款率 | 订单系统 | 现货订单退款数 / 现货订单总数 × 100% |
| 预售退款率 | 订单系统 | 预售订单退款数 / 预售订单总数 × 100% |
| 现货库存 | 聚水潭 ERP | 实时可用库存 |
| 采购在途 | 聚水潭 ERP | 已下单未入库的采购数量 |

### 9.2 计算规则

#### SPU 维度计算

- **SPU 销售金额** = Σ(SPU 下所有 SKU 的销售金额)
- **SPU 退款率** = Σ(各 SKU 退款金额 × SKU 退款率) / Σ(各 SKU 退款金额)（按金额加权平均）
- **SPU 库存** = Σ(SPU 下所有 SKU 的现货库存)

#### SKU 维度计算

- **SKU 销售金额** = 各渠道销售金额之和
- **SKU 实付金额** = 各渠道实付金额之和
- **总退款率** = 售前退款率 + 售后退款率（直接相加）

#### 通用计算

- **排名** = 按销售金额降序排列后的序号（受筛选条件影响）

### 9.3 数据更新频率

| 数据类型 | 更新频率 | 说明 |
|----------|----------|------|
| 销售数据 | 每小时 | 从订单系统同步 |
| 退款率 | 每小时 | 随销售数据同步计算 |
| 库存数据 | 每30分钟 | 从聚水潭 ERP 同步 |
| 历史均值（退款率阈值） | 每日凌晨 | 计算最近30天均值 |

## 十二、待确认事项

> 以下事项需与业务方进一步确认：

1. **SPU 维度搜索字段**：除了抖音ID和货号，是否需要支持按商品名称模糊搜索？
2. **SKU 维度搜索字段**：除了商家编码，是否需要支持按颜色搜索？

4. **退款率计算口径**：SPU 维度的退款率按金额加权平均是否合理？
5. **历史均值范围**：退款率阈值基于最近30天均值，是否需要支持自定义天数？
6. **库存数据权限**：库存数据是否需要对特定角色隐藏？
7. **时间筛选范围**：「自定义」日期范围是否有最大跨度限制（如最多90天）？
8. **数据量上限**：SKU 数量增长后（如超过500个），是否需要分页或虚拟滚动？
9. **商品图片**：正式环境的商品主图来源（CDN 地址规则）？
