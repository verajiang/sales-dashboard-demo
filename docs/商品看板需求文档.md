## 一、模块概述

### 1.1 模块定位

商品看板是「全渠道销售综合看板」系统的子模块，以 **SKU 维度**（即每个颜色/规格为一个独立行）展示全渠道商品销售数据，帮助运营和供应链团队快速掌握各 SKU 的销售表现、退款健康度及库存状况。

### 1.2 核心价值

- **SKU 粒度分析**：下钻到颜色/规格级别，而非 SPU 汇总，精准定位问题 SKU
- **全渠道对比**：同一 SKU 在抖音、天猫、微信各渠道的销售和退款一目了然
- **库存联动**：现货库存 + 采购在途，辅助供应链决策
- **退款预警**：通过颜色标记快速识别退款率异常的 SKU

### 1.3 目标用户

| 角色 | 使用场景 |
|------|----------|
| 运营经理 | 监控 SKU 销售排名、退款率趋势，优化选品和定价策略 |
| 供应链专员 | 关注库存预警，安排补货和采购计划 |
| 渠道运营 | 对比各渠道表现，调整投放和运营策略 |
| 数据分析师 | 导出数据做深度分析 |

### 3.4 退款率颜色阈值规则（基于历史均值动态校准）

退款率通过圆角标签（badge）展示，颜色阈值基于**历史均值动态计算**，而非固定数值。

#### 计算逻辑

系统每日自动计算最近 30 天的历史均值作为基准：

```
历史均值 = Σ(每日退款率) / 30 天

阈值计算：
- 预警下限 = 历史均值 × 0.9
- 预警上限 = 历史均值 × 1.3
```

#### 颜色判定规则

| 条件 | 背景色 | 文字色 | 含义 |
|------|--------|--------|------|
| ≤ 预警下限 | 浅绿 bg-emerald-50 | 绿色 text-emerald-600 | 正常（优于历史均值） |
| 预警下限 < rate ≤ 预警上限 | 浅黄 bg-amber-50 | 黄色 text-amber-600 | 预警（接近或略高于历史均值） |
| > 预警上限 | 浅红 bg-rose-50 | 红色 text-rose-600 | 异常（显著高于历史均值） |

#### 各退款率类型独立计算

| 退款率类型 | 计算基准 | 说明 |
|------------|----------|------|
| 总退款率 | 最近30天总退款率均值 | 售前+售后退款率之和的历史均值 |
| 现货退款率 | 最近30天现货退款率均值 | 仅现货订单退款率的历史均值 |
| 预售退款率 | 最近30天预售退款率均值 | 仅预售订单退款率的历史均值 |

#### 示例

假设最近 30 天总退款率均值为 20%：
- 预警下限 = 20% × 0.9 = 18%
- 预警上限 = 20% × 1.3 = 26%

则判定规则为：
| 当前退款率 | 颜色 | 含义 |
|------------|------|------|
| ≤ 18% | 绿色 | 优于历史平均水平 |
| 18% ~ 26% | 黄色 | 接近历史平均水平 |
| > 26% | 红色 | 显著高于历史平均水平，需关注 |

> **注意**：历史均值每日凌晨自动更新，阈值随之动态调整。

### 3.5 库存颜色阈值规则

#### 现货库存

| 条件 | 文字色 | 含义 |
|------|--------|------|
| ≥ 50 | 绿色 text-emerald-500 | 充足 |
| 30 ≤ stock < 50 | 黄色 text-amber-500 | 偏低 |
| < 30 | 红色 text-rose-500 | 紧急 |

#### 采购在途

| 条件 | 文字色 | 含义 |
|------|--------|------|
| > 50 | 绿色 text-emerald-500 | 在途充足 |
| 30 < transit ≤ 50 | 黄色 text-amber-500 | 在途一般 |
| ≤ 30 | 灰色 text-slate-500 | 在途较少 |

## 四、展开行（渠道明细）

### 4.1 交互规则

- **触发方式**：点击 SKU 主行任意位置
- **展开图标**：排名徽章内显示 ▶（收起）或 ▼（展开）图标
- **展开内容**：该 SKU 在各渠道的销售明细（仅显示有销售记录的渠道）
- **无渠道数据**：如果 SKU 没有任何渠道数据，则不显示展开图标，点击无响应

### 4.2 展开行字段

展开行显示的字段与主行基本一致，**但库存相关字段（现货库存、采购在途）显示为 "-"**，因为库存不分渠道。

| 字段 | 展开行是否显示 | 说明 |
|------|----------------|------|
| 排名 | 不显示（空白） | - |
| SKU信息 | 替换为渠道标签 | 如「抖音」「天猫」等 |
| 销售金额 | ✅ 显示 | 该渠道的销售金额 |
| 销售件数 | ✅ 显示 | 该渠道的销售件数 |
| 实付金额 | ✅ 显示 | 该渠道的实付金额 |
| 实付件数 | ✅ 显示 | 该渠道的实付件数 |
| 售前退款率 | ✅ 显示 | 该渠道的售前退款率 |
| 售后退款率 | ✅ 显示 | 该渠道的售后退款率 |
| 总退款率 | ✅ 显示 | 该渠道的总退款率（售前+售后） |
| 现货退款率 | ✅ 显示 | 该渠道的现货退款率 |
| 预售退款率 | ✅ 显示 | 该渠道的预售退款率 |
| 现货库存 | ❌ 显示 "-" | 库存不分渠道 |
| 采购在途 | ❌ 显示 "-" | 库存不分渠道 |
| 渠道 | 不显示（空白） | - |

### 4.3 展开行样式

- 背景色：浅灰 `#f8fafc`
- 文字大小：比主行小一号（12px / text-xs）
- 各渠道行之间用细分割线分隔
- 退款率颜色规则与主行一致

## 八、数据来源与计算规则

### 8.1 数据来源

| 数据类别 | 数据来源系统 | 说明 |
|----------|-------------|------|
| 销售金额/件数 | 订单系统 | 按下单时间统计，含所有状态订单 |
| 实付金额/件数 | 订单系统 | 扣除已退款订单后的净额 |
| 售前退款率 | 订单系统 | 发货前退款订单数 / 总订单数 × 100% |
| 售后退款率 | 订单系统 | 发货后退款订单数 / 总订单数 × 100% |
| 现货退款率 | 订单系统 | 现货订单退款数 / 现货订单总数 × 100% |
| 预售退款率 | 订单系统 | 预售订单退款数 / 预售订单总数 × 100% |
| 现货库存 | 聚水潭 ERP | 实时可用库存 |
| 采购在途 | 聚水潭 ERP | 已下单未入库的采购数量 |

### 8.2 计算规则

- **总退款率** = 售前退款率 + 售后退款率（直接相加，非加权计算）
- **SKU 销售金额** = 各渠道销售金额之和
- **SKU 实付金额** = 各渠道实付金额之和
- **排名** = 按销售金额降序排列后的序号（受筛选条件影响）

### 8.3 数据更新频率

| 数据类型 | 更新频率 | 说明 |
|----------|----------|------|
| 销售数据 | 每小时 | 从订单系统同步 |
| 退款率 | 每小时 | 随销售数据同步计算 |
| 库存数据 | 每30分钟 | 从聚水潭 ERP 同步 |

## 十一、待确认事项

> 以下事项需与业务方进一步确认：

1. **退款率计算口径**：当前总退款率 = 售前 + 售后直接相加，是否需要改为按金额/件数加权计算？
2. **库存数据权限**：库存数据是否需要对特定角色隐藏？
3. **时间筛选范围**：「自定义」日期范围是否有最大跨度限制（如最多90天）？
4. **数据量上限**：SKU 数量增长后（如超过500个），是否需要分页或虚拟滚动？
5. **商品图片**：正式环境的商品主图来源（CDN 地址规则）？
6. **渠道扩展**：未来是否可能新增渠道（如小红书、京东）？数据结构是否需要预留扩展性？
